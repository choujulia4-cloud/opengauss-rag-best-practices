基于 openGauss 的 RetrievalQA / RAG 项目最佳实践报告（骨架式汇总）
1. 项目目标与评测范围

本项目围绕 openGauss + HuggingFace / Cohere embedding + 外部检索 + 普通生成模型 + BentoML 服务封装 的 RetrievalQA 架构展开，目标不是仅验证系统“能否跑通”，而是总结一套适用于 标准 Linux 开发环境 与 AArch64 / RK3568 边缘环境 的工程化最佳实践。原始知识库文本明确指出，该项目需要同时考虑文档预处理、embedding 生成、向量检索、服务封装与部署约束之间的联动关系，而不是只关注单一模块。

当前阶段的评测覆盖了三个层面：

chunking 层：比较 length / mark 两类切分策略及不同 max_chunk_size、overlap 配置的统计特性与稳定性。
retrieval 层：比较不同 chunk 配置与相似度算子在多类问题上的 top-1 / top-3 表现。
端到端 RAG 层：比较 retrieval coverage、answer correctness、groundedness、completeness、hallucination，以及 retrieval fault / generation fault / mixed fault 的分离结果。
2. 系统架构与评测原则
2.1 最终采用的系统理解方式

本项目将 RetrievalQA 系统拆解为以下模块：

文档切分（chunking）
embedding 生成
OpenGauss 向量存储与检索
prompt 构造
generator 生成
BentoML 服务封装

这种拆分方式符合原始知识库文本中的工程叙事：OpenGauss 提供向量存储与检索骨架，chunking 决定信息表达方式，embedding 决定语义映射方式，retrieval 参数决定上下文选择方式，BentoML 则作为服务层封装整个调用链。

2.2 核心评测原则

本项目采用的核心评测原则是：

将 retrieval evaluation 与 generation evaluation 分离。

原始文本明确指出：retrieval evaluation 关注数据库是否返回正确或足够有用的 chunk，而 generation evaluation 关注最终答案是否正确、grounded 且完整；如果不分离两者，就容易把 generator 的能力误判为 retrieval 的能力，或者反过来掩盖 retrieval 层缺陷。

3. Chunking 最佳实践总结
3.1 总体结论

当前资料支持以下稳定结论：

length 切分当前不适合作为主路线。
mark 100 10 可以基本排除。
当前主候选已收敛到 mark_150_10、mark_200_10、mark_200_50。
mark_200 系列整体比 mark_150 更稳定。
mark_200_50 是当前默认主配置；mark_200_100 更适合作为边界增强候选，而不是直接替代默认配置。

这些结论来自 chunk 统计结果与 retrieval 表现的双重支持。统计结果显示，mark_100_10 块过碎、过短块较多；mark_150 在 overlap 增大后出现超长块；mark_200 在多个 overlap 下保持更稳定的长度分布和更低的异常块比例。

3.2 核心表格：Chunk 配置收敛结论表
配置	当前定位	保留/排除	结论说明
length 系列	非主路线	排除为主	当前英文技术知识库下，整体不如 mark 稳定
mark_100_10	过碎配置	排除	过短块多，边界信息容易破碎
mark_150_10	小 chunk 对照组	保留	某些比较类/部署任务类问题 top-1 更直接
mark_200_10	200 系列基线	保留	整体稳定，适合作为对照基线
mark_200_50	默认主配置	保留	综合稳定性、上下文完整性与 retrieval 表现最均衡
mark_200_100	边界增强候选	条件保留	对少数跨列表/跨边界问题更有优势，但不构成全面替代 200_50 的证据
3.3 代表性现象

一个代表性现象是：在边界敏感实验中，“four main forms of deliverables” 这类跨列表问题，在更大 overlap 下更容易让 top-1 直接覆盖完整信息，而普通定义类问题在 200_50 与 200_100 之间差异不明显。这说明 大 overlap 的收益是局部有效，而不是普遍有效。

4. Retrieval 最佳实践总结
4.1 总体结论

当前 retrieval 层的主要结论如下：

在 HuggingFace embedding 路线下，<#> 是当前最值得继续推进的相似度算子。
<-> 明显弱于 <#>，不建议作为主算子继续推进。
<=> 介于两者之间，但整体不如 <#>。
定义类、作用类、直接原因类问题的 retrieval 已较稳定。
列表/枚举类、元数据字段类、推荐方案类的主要瓶颈并不只是算子，而是答案天然分散、跨 chunk 边界。

这一结论与已有 retrieval 实验一致：在 200 token 规模下，<#> 在 top-1、top-3 及比较类、部署类问题上更稳，而 <-> 在多类问题上更容易召回包装性句子而不是核心答案句。

4.2 核心表格：Retrieval 算子对比结论表
维度	<#>	<=>	<->	结论
top-1 命中	最优	中等	最弱	<#> 当前最佳
top-3 覆盖	最优	次之	最弱	<#> 更适合作为主算子
比较类问题	较强	中等	偏弱	<#> 优势更明显
推荐/部署类问题	较强	中等	偏弱	<#> 更不易完全 miss
列表/枚举类问题	全部偏弱	全部偏弱	全部偏弱	主要瓶颈在 chunk 边界与信息分布，不是算子本身
4.3 代表性现象

一个典型现象是：“two embedding sources”“metadata fields”“four deliverables” 这类问题，即使更换算子，top-1 仍经常 miss。这说明这类问题的主要困难不在向量相似度公式，而在于答案本身需要多个信息点整合。

5. End-to-End RAG 最佳实践总结
5.1 总体结论

当前端到端 RAG 层可以形成以下结论：

系统已经完成从检索到生成的完整闭环验证。
当前主瓶颈在 generator，而不是 retrieval 层完全不可用。
flan-t5-base 已证明“可跑通”，但不适合作为最终最佳生成方案。
prompt 调整有帮助，但改善幅度有限。
后续优化的核心方向应是：保留当前 retrieval 主线，优先升级 generator。

已有阶段结果显示，在当前配置下，<#> 的 retrieval coverage 已经达到可用水平，但端到端最终答案仍常出现“只答半句”“过于极简”“缺少关键点”的情况，这与纯 generation fault 占比较高的现象是一致的。

5.2 核心表格：端到端层面阶段结论表
维度	当前观察	阶段结论
Retrieval coverage	top-1 / top-3 已具备可用性，尤其是 <#> 路线较稳	检索层不是完全失效
Correctness	仍有明显不足	generator 影响显著
Groundedness	整体不差，幻觉率较低	主要问题不是胡编，而是答不全
Completeness	当前偏低	生成层完整性是核心瓶颈
Fault 分离	pure generation fault 占比较高	后续优先升级 generator
5.3 代表性现象

端到端评测中最关键的现象不是“模型乱编很多”，而是：

检索已拿回相关 chunk，但 generator 仍倾向输出极简答案或不完整答案。

这说明当前需要优先解决的是生成器的上下文利用能力与答案展开能力，而不是重新推翻检索链路。

6. 部署与工程化最佳实践总结
6.1 总体结论

原始知识库文本对部署侧已经给出了非常清楚的工程指导：

BentoML 适合作为服务封装层，而不是 embedding 模型本身。
RK3568 属于资源受限边缘设备，不适合同时承担重 embedding 与重 generation。
更现实的边缘部署方式是：将大规模离线预处理放在更强机器上，边缘端只负责在线检索与轻量响应。
最终推荐不是“把所有组件都塞进板子”，而是“形成最小、稳定、可复现的部署闭环”。

这些结论均可直接由原始知识库文本支持。

6.2 核心表格：部署层建议表
部署要素	最佳实践建议
BentoML 角色	作为应用服务层，封装 chunker、embedder、database client、retriever、generator 等组件
RK3568 定位	资源受限的边缘节点，不适合作为重模型全栈承载端
离线预处理	文档切分、批量 embedding 生成、批量入库应在更强机器完成
板端职责	在线 query 处理、向量检索、轻量响应组装
最小方案	OpenGauss + 轻量服务端点本地部署，大规模 embedding 在外部机器完成
6.3 代表性现象

原始文本已经明确提出：对 RK3568 来说，推荐的最小方案是本地部署 OpenGauss 与轻量服务端点，同时将大规模 embedding 生成放在独立机器上完成。这一点与当前项目的实际推进方向高度一致。

7. 当前阶段的推荐配置
7.1 默认推荐配置

当前阶段可以先给出如下默认推荐配置：

Chunk 配置：mark_200_50
Retrieval operator：<#>
Retrieval 模式：dense retrieval
Generator：当前以 flan-t5-base 作为 baseline；后续重点比较更强 instruct generator
服务封装层：BentoML
部署策略：开发机完成完整实验，边缘端保留“本地检索 + 轻量服务”最小闭环

这一配置是基于当前已有结果的阶段性最优折中，而不是最终不可变配置。

7.2 条件化推荐
若知识库以普通技术说明、定义说明、过程说明为主，默认推荐 mark_200_50。
若知识库中跨列表、跨段落、边界敏感内容占比较高，可进一步考虑 mark_200_100 作为边界增强候选。
若目标是继续提升端到端质量，应优先升级 generator，而不是重新推翻现有 retrieval 主线。
8. 当前阶段仍未完全闭环的部分

本报告为阶段性最佳实践汇总，仍有若干收尾实验可进一步增强结论力度：

8.1 top_k 收尾实验

当前仍需进一步明确 top_k=1 / 3 / 5 对不同问题类型的影响，尤其要观察列表/边界类问题是否从更大的 top_k 中显著受益。该需求在前期整理材料中已经被明确提出。

8.2 生成器上限测试

建议使用人工构造的 gold context，跳过 retrieval，直接比较不同 generator 的 correctness、completeness 与 faithfulness，以验证当前 generator 能力上限。该实验将帮助进一步确认“当前主瓶颈在 generator”这一判断。

8.3 overlap 的正式推荐边界

当前已有证据支持 mark_200_50 为默认主配置、mark_200_100 为边界增强候选，但若后续需要形成更正式的 overlap 推荐规则，可围绕边界敏感问题再做小规模定向补测。

10. 当前阶段总结

截至当前阶段，本项目已经形成一条较清晰的最佳实践主线：

以 mark_200_50 作为默认 chunk 配置；
以 <#> 作为当前 HuggingFace embedding 路线下最优先推进的相似度算子；
以 retrieval / generation 分离评测为核心原则；
以“generator 完整性不足”作为当前端到端性能的主要瓶颈判断；
以 BentoML 作为服务封装层；
以“开发机完成完整实验，RK3568 实现最小可部署闭环”作为现实部署路径。

这一主线已经足够支撑后续进一步形成完整的“结构化评论体系”和更正式的最佳实践报告。
