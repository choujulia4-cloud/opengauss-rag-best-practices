# 附录：证据目录与归档规则

本附录用于说明正式报告与工程总结所依赖的证据材料如何分类存放、如何与正文对应，以及哪些文件属于“核心证据”、哪些文件属于“阶段性工程尝试材料”。

## 1. 证据使用原则

1. 正文只引用已经形成稳定结论的材料。
2. 原始日志、评分表、阶段性整理稿、raw output 压缩包不直接平铺进入正文，而应作为附录或归档材料。
3. Docker / BentoML / compose 等工程文件可以证明“做过工程化尝试”与“组件定位”，但在当前项目中并不自动等于“最终最佳实践参数配置”。
4. `典型报错与修复经验（raw）.txt` 只作为候选素材池，必须经过筛选后才能进入正式报告。fileciteturn6file0

## 2. 建议的归档目录

### 2.1 `chunk/`
存放 chunk 统计、chunk 收敛、overlap 补测等材料。

### 2.2 `mixed/`
存放 retrieval-only 的混合比较文件，例如不同 chunk 配置与不同 operator 的对比。

### 2.3 `rag/`
存放单一端到端配置的原始输出，例如 flan、Phi 等模型在固定配置下的检索与生成日志。

### 2.4 `topk/`
存放 top-1 / top-3 / top-5 等固定主配置下的 top-k 对比原始输出。

### 2.5 `eval/`
存放纯评价/评分文件，例如阶段汇总、生成器对比、top-k 评分表、gold context 评分表等。

### 2.6 `review/`
存放报告骨架、阶段策略说明、阶段性方法论总结等二次整理稿。

### 2.7 `raw_output/`
存放未整理的终端输出、历史日志压缩包、原始记录等，用于追溯但不直接用于正文引用。

## 3. 正文与证据的映射建议

### 3.1 A 版正式报告
- 第 4 章 Chunking 最佳实践：优先引用 `split_chunk_results.txt` 及其对应整理稿。
- 第 5 章 Retrieval 最佳实践：优先引用 `retrieval_results.txt` 及其 operator / overlap 汇总。
- 第 6 章 End-to-End RAG：优先引用 `rag_results.txt` 与阶段评分表。
- 第 7 章 Gold Context：优先引用 `goldContext_results3.txt`、`query -goldContext.txt` 以及 gold context 对应评分文件。
- 第 8 章 部署与工程化：引用代码文件与工程文件，但表述要克制。
- 第 10 章 关键报错与修复经验：从 `典型报错与修复经验（raw）.txt` 中筛选。fileciteturn7file18

### 3.2 B 版工程总结
优先使用已经形成“阶段性硬结论”的材料，不展开 raw output 与阶段性部署细节。

## 5. raw 错误经验材料的使用边界

`典型报错与修复经验（raw）.txt` 的内容覆盖环境依赖、openGauss、RAG 模型切换与工程化部署多个方面，但正式写入时应只保留：
- 有普遍工程意义的问题
- 与当前主结论强相关的问题
- 经过多轮定位后形成清晰经验的问题

不建议把一次性现象或未完全复现的问题写成正式经验。fileciteturn6file0
