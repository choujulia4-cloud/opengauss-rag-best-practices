# 1. 项目目标（写作草稿）
围绕 openGauss + HuggingFace / DPR + RetrievalQA / RAG 主链，完成从文本切分、embedding 生成、向量检索到生成器比较的端到端验证，并在 chunk、检索算子、生成模型与部署分工方面形成阶段性最佳实践。

# 2. 系统实现

## 2.1 主链路
- 文档切分
- DPR embedding
- openGauss 向量写入与检索
- context 构造
- generator 生成回答

## 2.2 关键模块
- `TextChunker`
- `HFEmbedder`
- `OPController`
- `RagPipeline`
- `test_rag.py`

## 2.3 当前主技术路线
- chunk：mark-based
- embedding：DPR
- retrieval operator：`<#>`
- generator：Phi-3.5-mini-instruct

# 3. 已完成的关键实验

## 3.1 Chunk 收敛
### 可直接写进总结的结论
- `mark_200_50` 为当前默认主配置
- `mark_200_100` 作为边界增强候选
- `mark_150_10` 保留为小 chunk 对照组
- `length` 路线与 `mark_100_10` 不再作为主路线推进

### 面试表达建议
项目没有停留在“切分能跑通”，而是对不同 chunk 配置做了统计与 retrieval 双重比较，最终不是凭直觉，而是通过数据把主配置收敛到 `mark_200_50`。

## 3.2 Retrieval 收敛
### 可直接写进总结的结论
- `<#>` 在当前 DPR 路线下综合最优
- `<->` 明显偏弱
- `<=>` 处于中间
- 定义 / 作用 / 直接原因类问题较稳
- 列表 / 枚举 / 元数据类问题仍然偏难

### 面试表达建议
不是把相似度算子写死，而是把算子选择作为实验维度之一，通过问题类型与 top-k 表现来决定主算子。

## 3.3 生成器收敛
### 可直接写进总结的结论
- flan-t5-base 已完成闭环打通，适合作 baseline
- Phi-3.5-mini-instruct 明显改善了 correctness、groundedness 与 completeness
- 当前主瓶颈逐步转向生成层，而不是检索层完全不可用

### 面试表达建议
项目后期最关键的判断不是“继续盲目调检索”，而是识别出“检索命中了，但生成器没把答案说完整”，因此把优化重点转向更强 generator。

## 3.4 Gold Context 上限观察
### 可直接写进总结的结论
- gold context 结果没有达到预期上限
- 出现拒答、局部幻觉与完整性不足
- 说明当前生成器仍是核心瓶颈
- 同时也提示 prompt / 输入组织方式会显著影响结果

### 面试表达建议
项目没有把“给了完美 context 效果仍一般”简单解释为模型无用，而是进一步识别出模型能力边界与评测设置本身的影响。

# 4. 关键工程判断

## 4.1 Retrieval evaluation 与 generation evaluation 必须分离
- 否则容易把生成器能力误判为检索能力，或反之。

## 4.2 Chunk / operator / generator 必须解耦评测
- 否则很难知道到底是哪一层在决定结果。

## 4.3 完美检索不等于完美答案
- gold context 结果说明生成器利用上下文的能力本身也是瓶颈。

## 4.4 边缘板端应追求最小闭环，而不是照搬开发机全栈
- 更现实的路径是“本地检索 + 轻量服务 + 外部重计算协同”。

## 4.5 工程实现比单次跑通更重要
- 通过阶段性评测、故障分类与配置收敛，项目逐步从“实验 demo”转向“可复用工程经验”。

# 可直接用于简历 / 面试的总结话术（草稿）
本项目围绕 openGauss + DPR + RAG pipeline 完成了从文本切分、embedding 写库、向量检索到生成器对比的端到端验证，并基于多轮实验将主配置收敛为 `mark_200_50 + <#> + DPR + Phi-3.5-mini-instruct`。在实验过程中，我将 retrieval 与 generation 分离评测，发现系统瓶颈并不只是检索命中率，而更多体现在生成器对上下文的利用和答案完整性上；同时结合边缘部署约束，形成了“开发机完成重处理、板端保留本地检索与轻量服务最小闭环”的工程判断。
