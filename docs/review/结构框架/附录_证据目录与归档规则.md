# 附录：证据目录与归档规则

---

## 1. 目的
本文件用于说明正式报告与工程总结所依赖的证据材料如何分层存放、如何引用以及哪些材料属于“正文可直接下结论”的主证据，哪些材料只适合作为补充附件。

---

## 2. 归档原则

### 2.1 正文只保留四类内容
- 结论性描述
- 核心表格
- 核心指标
- 少量代表性现象

### 2.2 原始证据统一放附录或归档目录
包括但不限于：
- chunk 统计原始输出
- retrieval top-k 原始输出
- 端到端 RAG 原始输出
- gold context 原始输出
- 人工评分表
- 阶段性日志与终端输出
- 配置文件与代码文件
- 原计划 / 策略说明 / 归档说明

### 2.3 证据优先级
建议按以下优先级组织：

#### 一级主证据
可直接支撑正文关键结论的文件：
- `split_chunk_results.txt`
- `retrieval_results.txt`
- `rag_results.txt`
- gold context 相关结果文件
- 各阶段评分文件
- `query整合.txt`
- `query -goldContext.txt`

#### 二级实现证据
支撑系统架构与模块实现的文件：
- `text_chunker.py`
- `hf_embedder.py`
- `op_controller.py`
- `rag_pipeline.py`
- `test_rag.py`

#### 三级阶段性工程材料
用于说明实现路径与部署尝试，但不直接作为“最终推荐配置”的证据：
- `bentofile.yaml`
- `docker-compose.yml`
- `Dockerfile-cpu`
- `init_db.sh`
- `service.py`

#### 四级方法论与阶段建议材料
用于说明项目推进思路，但不宜直接当作最终实证：
- `最佳实践报告骨架式汇总.txt`
- `阶段性评测结果.txt`
- `典型报错与修复经验（raw）.txt`

## 4. 正文引用建议

### 4.1 Chunking 章节建议引用的主证据
- chunk 统计原始文件
- overlap 边界补测文件
- 混合配置统计汇总文件

### 4.2 Retrieval 章节建议引用的主证据
- retrieval operator 对比文件
- top-1 / top-3 命中比较
- 代表性 query 的 top-k 输出

### 4.3 RAG 章节建议引用的主证据
- flan vs Phi 对比文件
- top-k 对比文件
- 端到端评分文件
- 典型答案样例

### 4.4 Gold Context 章节建议引用的主证据
- gold context 原始输出
- gold context 评分文件
- 对照 query 与 gold context 文本

### 4.5 工程实现章节建议引用的主证据
- 核心代码文件
- OpenGauss 初始化脚本
- BentoML / Docker / compose 文件（仅用于说明实现路径，不作为最终配置证据）

## 5. 正文与附录的边界

### 5.1 应写进正文的内容
- 收敛结论
- 推荐配置
- 指标对比
- 故障分类结果
- 关键工程判断

### 5.2 不应直接堆进正文的内容
- 大段终端输出
- 全量 query 原始答案
- 全量 top-k 原始内容
- 重复性的报错记录
- 所有阶段性想法与未筛选经验

---

## 6. 对“典型报错与修复经验（raw）.txt”的使用原则
该文件只作为“候选素材池”，不应原样照搬进最终报告。建议处理方式如下：

1. 优先筛选：
   - 多轮定位才解决的问题
   - 在 openGauss / RAG / HuggingFace / Docker / AArch64 中具有典型性的坑
   - 能转化成可复用工程经验的问题

2. 剔除：
   - 重复问题
   - 偶发性、只影响单次实验的问题
   - 不能支撑项目主线判断的问题

3. A / B 两版使用差异：
   - A 版可保留更完整的问题分层与经验总结
   - B 版只保留最能体现工程判断的 3–5 条

---

## 7. 对 BentoML / Docker / compose 文件的使用原则
这些文件可以作为以下内容的证据：
- 服务封装方向
- 容器化尝试
- 多架构 / 边缘部署尝试
- 工程实现路径

但不应直接用作以下结论的唯一证据：
- 最终主配置
- 最终最佳部署参数
- 最终最优服务形态

---

## 8. 建议的“正文-证据”映射关系

| 正文章节 | 主要证据类型 | 备注 |
|---|---|---|
| Chunking 最佳实践 | chunk 统计、边界补测 | 主证据 |
| Retrieval 最佳实践 | retrieval 对比、top-k 输出 | 主证据 |
| End-to-End RAG | flan / Phi 对比、评分表 | 主证据 |
| Gold Context 异常 | gold context 输出与评分 | 主证据 |
| 系统架构与实现 | 核心代码文件 | 主证据 |
| 部署与工程化 | Docker / BentoML / compose | 说明性证据 |
| 关键报错与修复经验 | raw 经验文件 + 原始日志 | 候选素材，需筛选 |
| 原计划与差异说明 | 申请书 + 骨架 + 阶段性材料 | 说明性证据 |

---

## 9. 归档时的操作建议
- 优先保留“一个结论对应一组原始证据”的映射关系。
- 对拆分文件统一编号。
- 在同一配置存在多个版本时，保留版本号而不是手动覆盖。
- 工程实现文件与评测文件分目录存放，避免混淆。
